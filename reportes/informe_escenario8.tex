\section{Introducci\'on}
El presente informe documenta el \textbf{Escenario 8} del taller, cuyo prop\'osito es construir un \textbf{pipeline automatizado en Python} que calcule las cinco m\'etricas ISO/IEC 25022 del cat\'alogo de MediSalud (Escenario 6), partiendo de los datos validados en el Escenario 7, y que genere un archivo de indicadores en formato JSON reutilizable para tableros de control. Adem\'as, se integr\'o este pipeline en un flujo de \textbf{Integraci\'on Continua} mediante GitHub Actions.

\subsection{Objetivo del escenario}
Transformar las f\'ormulas normativas abstractas de ISO/IEC 25022 en \textbf{c\'odigo ejecutable, reproducible y auditable}, de modo que el c\'alculo de m\'etricas pueda ejecutarse de forma peri\'odica y autom\'atica, eliminando la dependencia de c\'alculos manuales propensos a errores.

% ============================================================
\section{Arquitectura del Pipeline de Medici\'on}

El pipeline sigue una arquitectura ETL (\textit{Extract--Transform--Load}) de tres etapas, implementada \'integramente en Python con la biblioteca Pandas:

\begin{enumerate}
    \item \textbf{Extracci\'on (Extract):} El m\'odulo \texttt{metricas\_iso25022.py} importa los tres archivos CSV validados (\texttt{logs\_hce.csv}, \texttt{encuesta\_satisfaccion.csv}, \texttt{incidentes\_2025\_iso\_25022.csv}) mediante \texttt{pd.read\_csv()}.

    \item \textbf{Transformaci\'on (Transform):} Funciones independientes procesan los DataFrames para calcular cada m\'etrica:
    \begin{itemize}
        \item \texttt{metrica\_efectividad()}: $X = A/B$ (notas completadas / intentadas)
        \item \texttt{metrica\_eficiencia()}: $\bar{t} = \sum t_i / n$
        \item \texttt{metrica\_satisfaccion()}: $X = \sum s_i / (N \times 5)$
        \item \texttt{metrica\_libertad\_riesgo()}: $X = F_e / F_t$
        \item \texttt{metrica\_eficiencia\_por\_sede()}: desagregaci\'on para Cobertura de Contexto
    \end{itemize}
    Cada funci\'on retorna un diccionario con: \texttt{valor}, \texttt{unidad}, \texttt{umbral} y \texttt{cumple} (booleano).

    \item \textbf{Carga (Load):} El script \texttt{exportar\_reporte.py} empaqueta todos los resultados en un archivo \texttt{dashboards/indicadores.json}, consumible por herramientas de visualizaci\'on (Grafana, Power BI, etc.).
\end{enumerate}

\section{Integraci\'on Continua (CI)}

Para convertir el programa de medici\'on en un proceso continuo y reproducible, se configur\'o un \textit{workflow} en GitHub Actions (\texttt{.github/workflows/medicion\_calidad.yml}):

\begin{itemize}
    \item \textbf{Trigger:} Ejecuci\'on programada cada lunes a las 06:00 UTC v\'ia \texttt{cron}, o manual v\'ia \texttt{workflow\_dispatch}.
    \item \textbf{Entorno:} Ubuntu con Python 3.11 y dependencias (pandas, numpy).
    \item \textbf{Pasos:} Checkout del repositorio $\rightarrow$ Instalaci\'on de dependencias $\rightarrow$ Ejecuci\'on de \texttt{metricas\_iso25022.py} $\rightarrow$ Exportaci\'on a JSON $\rightarrow$ Subida del artefacto \texttt{indicadores.json}.
\end{itemize}

\section{Preguntas de Discusi\'on}

\begin{enumerate}
    \item \textbf{?`Por qu\'e es importante que el c\'alculo de m\'etricas sea autom\'atico y no manual?}

    Porque un programa de medici\'on que depende de c\'alculos manuales es: (a) \textbf{propenso a errores} (transcripci\'on, f\'ormulas mal aplicadas), (b) \textbf{no escalable} (con 3\,150 registros es manejable, pero con millones de registros de producci\'on es inviable), y (c) \textbf{no reproducible} (distintas personas pueden aplicar criterios distintos de limpieza o agregaci\'on). La automatizaci\'on garantiza que el mismo c\'odigo, con los mismos datos, produce siempre el mismo resultado, lo cual es un requisito fundamental de la evaluaci\'on de calidad seg\'un ISO/IEC 25040.

    \item \textbf{?`Qu\'e ventaja aporta GitHub Actions frente a ejecutar el script manualmente cada semana?}

    GitHub Actions garantiza que la medici\'on se ejecuta de forma \textbf{puntual e independiente} del factor humano. Si un ingeniero olvida ejecutar el script o est\'a de vacaciones, la medici\'on igual se realiza. Adem\'as, el historial de ejecuciones queda registrado como artefactos del repositorio, proporcionando \textbf{trazabilidad auditora} completa.

    \item \textbf{?`C\'omo interpret\'o la gerencia el hecho de que la Eficiencia global (7.43s) cumple el umbral, pero la Satisfacci\'on (0.71) no?}

    Este resultado ilustra una situaci\'on en la que el sistema es t\'ecnicamente eficiente \textbf{en promedio}, pero los usuarios m\'as cr\'iticos (m\'edicos) perciben una mala experiencia. El promedio global enmascara la degradaci\'on en horas pico, que es precisamente cuando los m\'edicos m\'as usan el sistema. La soluci\'on no es solo optimizar el tiempo promedio, sino garantizar que el \textbf{percentil 90} del tiempo de registro tambi\'en cumpla el umbral, como exige literalmente el RNF-01.
\end{enumerate}

\begin{itemize}

    \item El pipeline revel\'o que \textbf{2 de 5 m\'etricas no cumplen} sus umbrales (Satisfacci\'on y Libertad de Riesgo), proporcionando evidencia cuantitativa para priorizar acciones de mejora.
    \item La integraci\'on con \textbf{GitHub Actions} convierte el programa de medici\'on en un proceso continuo y reproducible, sentando las bases para la construcci\'on de indicadores y dashboards del Escenario 9.
    \item Este hito es fundamental para que MediSalud abandone la gesti\'on basada en ``percepciones'' y transite hacia una \textbf{gobernanza de calidad basada en datos observables}.
\end{itemize}
